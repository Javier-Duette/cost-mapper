# ADR-015 — Validación de compatibilidad de unidades en el mapper IFC

**Estado:** Aceptado
**Pregunta original:** ¿El mapper debe impedir asignar ítems del catálogo cuya unidad no es
compatible con la unidad de medición del tipo de elemento IFC?

## Contexto

El módulo budget calcula cantidades geométricas asumiendo unidades específicas por tipo IFC
(IfcWall → m², IfcColumn → m o m³, IfcDoor → un, etc.). Si el mapper permite asignar un ítem
con unidad incompatible (ej: kg a un IfcWall), el presupuesto resultante es matemáticamente
incorrecto aunque no explote.

La lógica de qué unidades aplican a cada tipo IFC ya existía en `budget/service.py` (`_QTO_MAP`).

Alternativas consideradas:
1. Warning en frontend (no bloquear) — descartado: silencia errores de datos que afectan
   el presupuesto sin aviso claro.
2. Campo `unit` en `ifc_elements` (persistir en DB) — descartado: es información derivable
   del `ifc_type`, no hay razón para persistirla.
3. Flag `force: bool` para saltear la validación — descartado para MVP; el usuario que
   necesite asignar un tipo no-estándar puede editar el catálogo.

## Decisión

- El mapa `ifc_type → unidades válidas` vive en `backend/ifc_units.py` (módulo compartido,
  sin tabla DB). Lo importan mapper y cualquier módulo futuro que lo necesite.
- El backend valida en `create_assignment()` y `assign_group_manual()`: si la unidad del ítem
  no está en el conjunto compatible para ese tipo IFC, retorna HTTP 422 con mensaje descriptivo.
- Las sugerencias (`get_suggestions()`) se filtran por unidad compatible antes de retornarlas.
- Tipos IFC desconocidos (no en el mapa, ej: IfcSpace, IfcSite) no tienen restricción — todos
  los ítems son asignables.
- El API expone `compatible_units: list[str] | null` en el contexto de sugerencias para que
  el frontend pueda mostrar ítems incompatibles como deshabilitados (con tooltip explicativo).

## Consecuencias

- Asignaciones incoherentes (kg a IfcWall) quedan bloqueadas en backend.
- Las sugerencias automáticas solo proponen ítems medibles con las unidades correctas.
- El frontend muestra el motivo del bloqueo en lugar de simplemente fallar el request.
- Si en el futuro se agregan tipos IFC nuevos al presupuesto, hay que agregar la fila
  correspondiente en `backend/ifc_units.py`.
