# ADR-006 — Extensibilidad y documentación para desarrolladores

**Estado:** Aceptado  
**Pregunta original (nueva):** ¿Cómo hacemos que el programa sea fácil de modificar para estudiantes y desarrolladores? ¿Es suficiente con buena documentación? ¿Ayudaría una biblioteca de skills para agentes de programación?

---

## Contexto

El Perfil 3 (estudiante/profesional en formación) no solo consume el software: quiere entenderlo, modificarlo y usarlo como base para sus propias ideas. Open source sin buena documentación de desarrollo es open source de nombre. Un proyecto que los estudiantes pueden modificar con confianza tiene un impacto educativo real en el mercado paraguayo de BIM.

Además, la proliferación de agentes de IA para programación (Claude Code, GitHub Copilot, Cursor) significa que la documentación técnica es consumida tanto por humanos como por máquinas.

## Decisión

**Documentación técnica como entregable de primer nivel:**

- `CONTRIBUTING.md`: Guía de contribución.
- `ARCHITECTURE.md`: Diagrama del sistema y explicación de cada módulo.
- `docs/adrs/`: El historial de decisiones de diseño como documentación técnica fundamental.
- Docstrings en todo el código (Google style para Python, JSDoc para TS).
- Un `CLAUDE.md` en la raíz con instrucciones para agentes de IA.

**Biblioteca de skills para agentes de programación:**

El repositorio incluirá una carpeta `ai-skills/` con prompts especializados para agentes de IA. Los skills se crean junto con la ejecución del código — cuando se implementa una funcionalidad que tiene sentido documentar como skill reutilizable, se agrega en ese momento. No se pre-definen skills que aún no existen.

**Arquitectura modular como requisito de diseño:**

Módulos con responsabilidades claras y contratos definidos, para que un estudiante pueda modificar un módulo sin necesitar entender todos los demás.

### Continuidad entre agentes de IA

El sistema completo tiene cinco piezas que trabajan juntas:

1. **`CLAUDE.md`** — el punto de entrada del agente (briefing operativo)
2. **ADRs** — la memoria de decisiones (por qué las cosas son así)
3. **`DEVLOG.md`** — el diario de sesión (qué pasó ayer)
4. **Commits con contexto** — el qué y el por qué en cada cambio
5. **Documentación-primero** — la secuencia siempre es: ADR → DEVLOG → código → commit

## Consecuencias

- El `CLAUDE.md` se crea antes del primer commit de código.
- El `DEVLOG.md` se inicia con la primera entrada de datos y se actualiza en cada sesión.
- La carpeta `ai-skills/` se construye junto con el código.
- Las interfaces entre módulos deben estar explícitamente documentadas.
- Cambiar de agente de IA no requiere re-explicar el proyecto.
