# ADR-009 — Migración del backend a SQLModel

**Estado:** Propuesto  
**Pregunta original:** ¿Deberíamos reemplazar SQLAlchemy puro con SQLModel para reducir duplicación de modelos entre la base de datos y las validaciones de Pydantic?

---

## Contexto

El backend de Cost-Mapper utiliza actualmente **SQLAlchemy ORM** para la persistencia en PostgreSQL. En paralelo, los endpoints de FastAPI validan las solicitudes con esquemas **Pydantic** que frecuentemente espejean la estructura de los modelos SQLAlchemy:

```python
# backend/catalog/models.py
class CatalogItem(Base):
    __tablename__ = "catalog_items"
    id: Mapped[str] = mapped_column(String(36), primary_key=True)
    description: Mapped[str] = mapped_column(String(500), nullable=False)
    nbr_code: Mapped[str] = mapped_column(String(20), nullable=False)
    unit: Mapped[str] = mapped_column(String(50), nullable=False)
    unit_price: Mapped[Decimal] = mapped_column(Numeric(12, 2), nullable=False)
    # ... más campos

# backend/catalog/schemas.py
class CatalogItemResponse(BaseModel):
    id: str
    description: str
    nbr_code: str
    unit: str
    unit_price: Decimal
    # ... más campos
```

Esta duplicación:

1. **Aumenta la carga de mantenimiento.** Cada cambio en la estructura de datos requiere actualizar dos archivos. Una divergencia accidental entre los modelos causa errores silenciosos en validación.
2. **Viola DRY (Don't Repeat Yourself).** Los tipos y constrains se definen dos veces con riesgos de inconsistencia.
3. **Ralentiza la refactorización.** Renombrar una columna requiere buscar y actualizar referencias en múltiples lugares.

**SQLModel** es una librería desarrollada por el mismo autor de FastAPI (Sebastián Ramírez) que **combina SQLAlchemy y Pydantic en un único modelo**. Permite definir una tabla y un esquema de validación desde el mismo código, eliminando la duplicación.

---

## Alternativas evaluadas

### Opción A: Mantener SQLAlchemy + Pydantic (estado actual)

**Pros:**
- Cero cambios de código. El proyecto funciona como está.
- Separación explícita entre persistencia y validación.
- Amplio ecosistema de herramientas y documentación.

**Contras:**
- Duplicación de modelos.
- Mayor complejidad operativa.
- Más propenso a bugs por inconsistencias.

### Opción B: Migración a SQLModel

**Pros:**
- **Un único modelo que es válido para SQLAlchemy y Pydantic.** No hay duplicación.
- Menos código, más fácil de mantener.
- Mejor integración con FastAPI — los tipos de Pydantic se generan automáticamente.
- El API documentation (Swagger/OpenAPI) se actualiza automáticamente.
- Excelente documentación oficial y comunidad activa.
- **Compatible con async SQLAlchemy** — necesario para operaciones no-bloqueantes en FastAPI.

**Contras:**
- Requiere refactorización de toda la capa de modelos.
- Curva de aprendizaje — mezcla conceptos de SQLAlchemy, Pydantic y FastAPI.
- Menor cantidad de preguntas en StackOverflow comparado con SQLAlchemy puro (aunque está bien documentado).
- Complejidad en modelos complejos con relaciones profundas (aunque está bien documentado).

### Opción C: Usar TypedDict + SQLAlchemy

**Pros:**
- Sin dependencias externas nuevas.
- Python type-safe sin Pydantic.

**Contras:**
- No hay validación automática como en Pydantic.
- FastAPI no lo soporta directamente.
- Duplicación parcial que no resuelve el problema.

---

## Decisión

**Se adopta SQLModel como la solución estándar para el backend de Cost-Mapper a partir del ADR-009.**

Razón principal: En un proyecto educativo de código abierto, la **reducción de complejidad y duplicación** es crítica para que estudiantes y desarrolladores nuevos puedan entender y modificar el código. SQLModel logra esto sin sacrificar funcionalidad ni performance.

### Implementación

**Estructura de módulos (ajustada para SQLModel):**

Cada módulo en `backend/<modulo>/` tendrá la siguiente estructura:

```
models.py       ← modelos SQLModel (heredan de SQLModel, no de Base)
router.py       ← rutas FastAPI, no importa models para validación
service.py      ← lógica de negocio, llama a repository
repository.py   ← queries DB, sin lógica de negocio
# schemas.py    ← ELIMINADO — los modelos SQLModel sirven como esquema
```

Nota: El archivo `schemas.py` se elimina. Las validaciones request/response usan directamente los modelos SQLModel o subclases especializadas (ej: `CatalogItemCreate`, `CatalogItemUpdate`) si se requiere campos opcionales o parciales.

**Ejemplo de migración:**

```python
# Antes (SQLAlchemy + Pydantic separados)
# models.py
class CatalogItem(Base):
    __tablename__ = "catalog_items"
    id: Mapped[str] = mapped_column(String(36), primary_key=True)
    description: Mapped[str] = mapped_column(String(500), nullable=False)

# schemas.py
class CatalogItemResponse(BaseModel):
    id: str
    description: str

---

# Después (SQLModel)
# models.py
class CatalogItem(SQLModel, table=True):
    __tablename__ = "catalog_items"
    id: str = Field(primary_key=True, max_length=36)
    description: str = Field(max_length=500)
    # ... el mismo modelo es válido para DB y API responses

# router.py (FastAPI lo valida automáticamente)
@router.get("/items/{item_id}", response_model=CatalogItem)
def get_item(item_id: str, session: Session = Depends(get_session)):
    return repository.get_item_by_id(session, item_id)
```

**Validación y tipos:**

SQLModel permite especificar restricciones usando `Field()`:

```python
class APUComponent(SQLModel, table=True):
    __tablename__ = "apu_components"
    id: str = Field(primary_key=True)
    quantity: Decimal = Field(gt=0)  # mayor que 0
    unit: str = Field(max_length=50)
```

**Para modelos parciales** (crear/actualizar con campos opcionales):

```python
# Todas las validaciones en un lugar
class CatalogItemCreate(SQLModel):
    """Request body para crear un ítem (sin id, autogenerado)."""
    description: str = Field(max_length=500)
    nbr_code: str = Field(max_length=20)
    unit: str = Field(max_length=50)

class CatalogItem(SQLModel, table=True):
    """Modelo persistido en DB — hereda de SQLModel con table=True."""
    id: str = Field(primary_key=True)
    description: str
    nbr_code: str
    unit: str
```

### Migración incremental

La migración no es un cambio de "big bang":

1. **Fase 1:** Actualizar el módulo `catalog/` a SQLModel como piloto.
2. **Fase 2:** Evaluar problemas, ajustar el patrón si es necesario.
3. **Fase 3:** Migrar los demás módulos (`ifc_importer/`, `budget/`, `mapper/`, `exporter/`).

Esto permite descubrir incompatibilidades o limitaciones antes de comprometer todos los módulos.

### Compatibilidad con reglas de módulo

**Las reglas de módulo se mantienen intactas:**

- Ningún módulo escribe en la tabla de otro — sigue siendo responsabilidad de `service.py` y `repository.py`.
- `schemas.py` se elimina como archivo pero la validación se sigue haciendo en nivel de modelo — es más fuerte que antes porque toda validación está en un lugar.
- Las interfaces entre módulos no cambian — es una refactorización interna.

### Dependencias

```bash
# Agregar a requirements.txt
sqlmodel==0.0.16  # O la última version compatible
sqlalchemy>=2.0   # SQLModel depende de SQLAlchemy 2.0+
```

SQLModel es estable (usa versiones pinned) y tiene < 10 dependencias transitivas. No añade complejidad al despliegue.

## Consecuencias

**Positivas:**

- Cada estructura de datos se define **una única vez**, reduciendo bugs por inconsistencia.
- El código es más compacto y más fácil de leer. Un estudiante entiende el modelo de datos más rápido.
- Los tipos de Pydantic y las restricciones de BD están en el mismo lugar — cambiar uno automáticamente se refleja en el otro.
- FastAPI genera documentación OpenAPI más precisa porque los tipos Pydantic se obtienen directamente del modelo.
- Migración limpia — no hay código heredado que conviva con dos estilos diferentes.

**Negativas:**

- Requiere refactorización de toda la capa de modelos (alcance mediano, ~1-2 sesiones de trabajo).
- Algunos desarrolladores pueden encontrar la mezcla SQLModel+SQLAlchemy+Pydantic conceptualmente compleja al inicio.
- Menos ejemplos públicos en StackOverflow comparado con SQLAlchemy puro (pero la documentación de SQLModel es excelente).

**Sobre versiones de Python:**

SQLModel requiere Python 3.7+. Cost-Mapper ya especifica Python 3.11+, así que no hay limitaciones.

**Sobre compatibilidad con migraciones Alembic:**

Alembic funciona igual con SQLModel — la tabla generada es idéntica a la de SQLAlchemy puro. Ningún cambio en el pipeline ETL ni en las migraciones.
