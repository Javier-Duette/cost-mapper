## ADR-014 — IFC-first para mapeo; keynotes como export opcional

**Estado:** Aceptado
**Pregunta original:** ¿Cómo sincronizamos Cost-Mapper con Revit (catalogación) sin bloquear el desarrollo del mapper?

### Contexto
La iteración de exportación de keynotes validó que Revit impone requisitos estrictos de formato (encoding, saltos de línea, jerarquías y nodos contenedores). Aunque es viable generar un archivo de keynotes compatible, usar keynotes como mecanismo primario de sincronización y catalogación introduce fragilidad operativa:

- Requiere disciplina de modelado y configuración consistente en Revit.
- Aumenta el costo de mantenimiento del “contrato” (formato/estructura) y la probabilidad de errores difíciles de depurar.
- Desvía el foco del objetivo inmediato: construir un mapper robusto para IFC, que es el canal vendor-neutral del sistema.

En paralelo, el flujo principal del producto está diseñado alrededor de IFC + Python (ifcopenshell) para extracción de información de elementos y cálculo de cantidades en runtime. Mantener el mapeo canónico en IFC reduce acoplamiento a Revit y habilita iteración más rápida en `ifc_importer/` y `mapper/`.

### Decisión
1. **IFC es la fuente canónica para el mapeo**: el contrato primario es `GlobalId` + señales de clasificación extraídas del IFC (Psets, propiedades y/o `IfcClassificationReference` cuando exista).
2. **Keynotes no se usan como canal primario de sincronización**: el módulo `library/` y el flujo de keynotes quedan **pausados como feature principal** y se consideran **export opcional/auxiliar** (post-MVP o cuando haya una necesidad clara de integración Revit).
3. **El objetivo inmediato del mapper** es resolver de forma estable `GlobalId → catalog_item_id` (catálogo), con reglas de prioridad y trazabilidad de la “fuente” de clasificación, sin depender de keynotes.

### Consecuencias
**Positivas:**
- Reduce fragilidad y dependencia de detalles de Revit en la arquitectura core.
- Acelera el desarrollo del mapper e IFC importer, alineado con el objetivo vendor-neutral.
- Permite soportar múltiples orígenes (Revit, Archicad, etc.) mientras exporten IFC con propiedades útiles.

**Negativas / tradeoffs:**
- Se pospone la ergonomía de “catalogación desde Revit” basada en keynotes.
- Puede requerir acuerdos mínimos de export IFC (parámetros a exportar como Psets) para obtener señales de clasificación de buena calidad.

**Pendiente:**
- Definir un conjunto mínimo de propiedades/Psets recomendados (MVP) para mejorar mapeo automático desde IFC.
- Documentar reglas de prioridad de mapeo (explícito > classification > heurística) y cómo se registra `classification_source`.
