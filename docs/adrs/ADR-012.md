## ADR-012 — Extracción del catálogo TCPO V15 con LLM Vision y detección local de tablas

**Estado:** Aceptado
**Pregunta original:** ¿Cómo extraer, traducir y cargar los ítems y APUs del PDF TCPO V15 de forma confiable y económica?

### Contexto

El PDF `538948707-TCPO-BIM-15-Edicao.pdf` (411 MB) es **rasterizado con OCR incrustado** — el texto extraído por herramientas estándar (`pdfplumber`, `pdftotext`) no reproduce la estructura de tabla correctamente (códigos NBR multi-línea, columnas de consumos, variantes de aproveitamento). ADR-002 estableció el PDF V15 como fuente primaria; ADR-003 planteó "3 scripts secuenciales" pero no especificó la estrategia de extracción para el caso rasterizado.

El catálogo tiene ~1.000 páginas con tablas muy estandarizadas:
- Cada composición está encuadrada en un rectángulo con borde grueso
- El encabezado contiene el código NBR (multi-línea), descripción y unidad
- La tabla interior tiene columnas fijas: Código | Descrição | Unid. | Consumos
- Los códigos de componentes también son multi-línea (ej. "2N 36 16 25\n12 15")
- Algunas tablas tienen múltiples columnas de Consumos (variantes del mismo servicio)

Enviar páginas completas a una API de visión es ineficiente: ~70% de cada página es márgenes, texto de capítulo y encabezados que no aportan datos.

**Alternativas consideradas:**
- **OCR clásico (Tesseract):** bajo costo pero frágil para reconstruir estructura de tabla, manejo pobre de códigos multi-línea. Requiere parser adicional complejo.
- **pdfplumber directo:** inútil para PDFs rasterizados; el texto extraído no mantiene la estructura tabular.
- **Enviar página completa a Gemini Vision:** funciona pero ~3× tokens innecesarios por página.
- **Detección local + crop → Gemini por tabla:** cada tabla = ~400×300 px, significativamente menos tokens que la página completa. Permite prompt específico por composición.

### Decisión

**Herramienta standalone en `scripts/etl_tcpo/`** — completamente separada de la aplicación Cost-Mapper (consistente con ADR-003). Flujo:

1. **Detección local (sin API):** renderizar la página como imagen con `pymupdf`, detectar rectángulos tabulares con `OpenCV` (contornos externos por área y proporción), recortar cada tabla como imagen PIL.

2. **Extracción+Traducción (Gemini Vision):** enviar cada tabla recortada a `gemini-2.0-flash` con un prompt estructurado que:
   - Indica el formato multi-línea de los códigos NBR
   - Solicita JSON con esquema fijo: `[{nbr_code, description_pt, description_es, unit, components:[...]}]`
   - Traduce PT→ES en el mismo llamado (español técnico latinoamericano)
   - Maneja variantes de consumos como ítems separados

3. **Validación local:** verificar formato de códigos NBR, unidades conocidas, quantidades numéricas positivas. Rechazar y loguear ítems con datos inválidos.

4. **Carga en DB:** insertar `catalog_items` con `is_work_item=True` (ADR-011). Los ítems TCPO del PDF no tienen `unit_price` al momento de la extracción — ese campo queda `NULL` hasta que se cruce con la lista de precios Mandu'a. Los componentes del APU se insertan en `apu_components`.

5. **Progress tracking:** `scripts/data/tcpo_progress.json` — registro por página con estado (`done`/`error`/`skipped`), cantidad de ítems extraídos y timestamp. Las páginas ya procesadas no se re-envían a la API.

### Interfaz de uso

```bash
cd scripts/etl_tcpo

# Procesar páginas específicas
python main.py run --pages 36-50

# Procesar páginas puntuales
python main.py run --pages 36,40,45

# Ver resumen de progreso
python main.py status

# Re-procesar página (ignora cache)
python main.py run --pages 36 --force

# Debug: guardar imágenes de tablas detectadas (no llama a API)
python main.py detect --pages 36 --save-crops
```

### Consecuencias

- El ETL TCPO pasa de "3 scripts secuenciales" (ADR-003) a una herramienta interactiva selectiva. ADR-003 queda vigente en su intención pero su implementación concreta la define este ADR.
- Los ítems cargados NO tienen precio al extraerse — el pipeline completo requiere un paso adicional de cruce con precios Mandu'a (script separado, post-MVP).
- La calidad de extracción depende de Gemini 2.0 Flash y la claridad de las tablas. Tablas con notas al pie complejas o variantes múltiples requieren revisión manual.
- Costo estimado: ~0.001 USD por tabla. Con ~3-4 tablas/página × 1.000 páginas = ~3-4 USD para el catálogo completo.
- El campo `parent_nbr_code` de los ítems insertados se puede derivar truncando el código NBR al nivel superior (los últimos 2 segmentos en cero indican el padre).
- `progress.json` se agrega a `.gitignore` — es estado local del desarrollador, no del repositorio.
