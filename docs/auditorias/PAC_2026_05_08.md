**Origen:** `docs/auditorias/AUDITORIA_2026_05_08.md`  
**Estado:** Propuesto para ejecución

**Criterio rector:** corregir primero lo que bloquea continuidad, build y coherencia arquitectónica; después limpiar deuda documental y UX.

---

## Decisiones cerradas por el usuario

1. **`AGENTS.md` será la entrada canónica para agentes.**  
   Motivo: el proyecto no depende de un único agente. `CLAUDE.md` puede conservarse como compatibilidad o ser reducido a puente, pero no debe seguir compitiendo como fuente principal.

2. **`is_verified=false` bloquea entregables por defecto.**  
   Aplica a PDF, Excel, informes IFC y keynotes. Excepción permitida: keynotes puede ofrecer una liberación manual porque usa código y descripción, no precios, coeficientes ni APUs. Esa excepción debe quedar explícita, auditada y acompañada por advertencia.

3. **`settings_users` es un catálogo transitorio pre-auth.**  
   No reemplaza al futuro modelo definitivo de usuarios. Debe adaptarse a las reglas del proyecto ahora, sin sobrediseñarlo como sistema final de autenticación.

4. **Keynotes Revit requieren validación técnica.**  
   La documentación oficial de Autodesk define keynotes como TXT tabulado y recomienda guardar como Unicode cuando se usan caracteres compatibles con Unicode. Para Cost-Mapper, el PAC asume export tabulado Unicode y validación con Revit real antes de cerrar el punto.

Fuentes consultadas:
- Autodesk Help 2024, “About Editing the Keynote File”: https://help.autodesk.com/cloudhelp/2024/ENU/RevitLT-DocumentPresent/files/GUID-28DE1430-4838-48E1-9228-335A1D80FC2D.htm
- Autodesk Help 2024, “Keynoting”: https://help.autodesk.com/cloudhelp/2024/ENU/Revit-HaveYouTried/files/GUID-4ADEA4F7-536C-4D1C-BEAE-47AFEBE9A038.htm

## Criterio de cierre del PAC

El PAC se considera cerrado cuando:

- `npm.cmd run build` pasa.
- Los tests backend disponibles pasan en el entorno local documentado.
- `AGENTS.md` queda como fuente canónica versionada.
- `settings` cumple la estructura de 4 archivos o tiene una excepción formal aceptada.
- La regla de `is_verified` está aplicada en backend para entregables.
- Keynotes fue probado en Revit o queda marcado explícitamente como pendiente técnico bloqueante.
- DEVLOG, ADRs y docs principales reflejan el estado real.
- Existe commit + push final conforme ADR-008.
